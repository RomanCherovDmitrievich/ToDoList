# Мобильная архитектура

## Почему мобильные клиенты отделены

Текущий проект — это десктопное JavaFX-приложение, работающее напрямую с SQLite/PostgreSQL.
Такой подход подходит для десктопа, но не безопасен для Android и iOS клиентов.

Мобильные клиенты не должны подключаться к PostgreSQL напрямую с учётными данными базы данных.
Чтобы сохранить аккаунты связанными и задачи синхронизированными между десктопом, Android и iPhone, нам нужен небольшой API синхронизации перед серверной базой данных.

## Как должен быть разделён текущий проект

1. Аутентификация:
   - вход по имени пользователя или email
   - регистрация
   - запрос сброса пароля
   - подтверждение сброса пароля
2. Домен задач:
   - DTO задачи
   - DTO сессии пользователя
   - перечисления для приоритета/категории
   - поля повторения/напоминания
3. Локальное хранилище:
   - Android: Room/SQLite
   - iOS: SwiftData или SQLite
4. Синхронизация:
   - отправка локальных изменений
   - получение серверных изменений
   - разрешение конфликтов по `updatedAt`
5. UI:
   - экран входа
   - большой календарь
   - задачи на выбранный день
   - настройки

## Рекомендуемый серверный контракт

Десктопное приложение уже содержит все необходимые поля в локальных моделях, поэтому оба нативных клиента должны использовать ту же JSON-структуру.

### Сессия пользователя

```json
{
  "userId": "uuid",
  "username": "roman",
  "email": "roman@example.com",
  "accessToken": "jwt-or-session-token",
  "refreshToken": "refresh-token"
}
```

### DTO задачи

```json
{
  "id": "uuid",
  "title": "Prepare report",
  "description": "Q1 summary",
  "startTime": "2026-03-25T10:00:00",
  "endTime": "2026-03-25T12:00:00",
  "priority": "IMPORTANT",
  "category": "WORK",
  "completed": false,
  "overdue": false,
  "createdAt": "2026-03-24T19:00:00",
  "updatedAt": "2026-03-25T08:10:00",
  "ownerUserId": "uuid",
  "sortIndex": 10,
  "recurrenceRule": "NONE",
  "reminderOffsetMinutes": 15,
  "recurrenceEnd": null,
  "deletedAt": null
}
```

## Предлагаемые эндпоинты

1. `POST /api/v1/auth/login`
2. `POST /api/v1/auth/register`
3. `POST /api/v1/auth/password/reset/request`
4. `POST /api/v1/auth/password/reset/confirm`
5. `GET /api/v1/tasks?updatedAfter=<timestamp>`
6. `POST /api/v1/tasks/sync`

## Правила offline-first

1. Читать сначала из локальной базы данных.
2. Сохранять изменения локально перед любым сетевым вызовом.
3. Добавлять каждое локальное изменение в очередь синхронизации.
4. Отправлять изменения из очереди, когда сеть доступна.
5. Хранить `ownerUserId`, `updatedAt` и `deletedAt` для каждой задачи.
6. Использовать `updatedAt` как правило разрешения конфликтов по умолчанию, если позже не будет добавлено более умное слияние.

## Текущее ограничение

Файлы в `mobile/android` и `mobile/ios` — это нативные стартовые наборы.
Они уже отражают текущие десктопные модели и экраны, но им всё ещё нужен реальный API синхронизации, прежде чем они смогут делиться живыми аккаунтами и задачами с десктопным приложением.
